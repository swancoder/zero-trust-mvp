==============================================================
CHAIN OF THOUGHT
==============================================================

CERTIFICATE HIERARCHY:
  ZTE-CA (self-signed, 1y root)
  ├── client.p12      → presented by gateway + service-a as CLIENT cert on outbound calls
  ├── service-a.p12   → service-a's SERVER cert (SANs: service-a, localhost)
  └── service-b.p12   → service-b's SERVER cert (SANs: service-b, localhost)
  truststore.p12      → CA cert only; used as trust anchor by all services

  One shared "internal client" cert is used for all outbound mTLS calls (MVP simplification).
  Production: each service has its own SPIFFE identity (SVID).

ON-BEHALF-OF TOKEN FLOW:
  1. User presents Keycloak JWT to gateway.
  2. ZteAuthorizationFilter checks DB policy (existing, runs first).
  3. UserContextPropagationFilter (order HIGHEST_PRECEDENCE+200) strips any incoming
     X-ZTE-User-Context header (prevents injection), extracts sub + realm_access.roles
     from the Keycloak JWT, creates a 30s-TTL HMAC-SHA256 JWT signed with ZTE_OBO_SECRET.
  4. Gateway forwards request to service-a over HTTPS/mTLS with:
       X-ZTE-User-Context: <signed OBO JWT>
       X-User-Id: <sub>
  5. Service A calls Service B over HTTPS/mTLS, forwarding X-ZTE-User-Context unchanged.
     Service A does NOT re-sign the token — it's a delegation chain, not re-issuance.
  6. Service B validates HMAC signature, checks expiry, returns user context.

WHY NOT FORWARD THE ORIGINAL KEYCLOAK JWT:
  Forwarding the Keycloak JWT to downstream services means any compromised service
  can use it to authenticate as the user to Keycloak or other services.
  The OBO token is gateway-signed, short-lived (30s), and contains only what downstream
  services need (sub, roles). A stolen OBO token has 30s of blast radius.

DYNAMIC SSLCONTEXT RELOADING:
  Client-side (outbound HTTPS): ReloadableSslContextFactory wraps an
  AtomicReference<io.netty.handler.ssl.SslContext>. Reactor Netty's
  HttpClient.secure(Consumer<SslContextSpec>) lambda is called per TCP connection.
  Since the lambda calls sslFactory.current() (reads the AtomicReference), new
  connections pick up the refreshed SslContext transparently. An @Scheduled task
  checks file modification time every 5 minutes and rebuilds if changed.

  Server-side: Spring Boot's embedded Netty (WebFlux) reads server.ssl.* at startup
  and builds the SslContext once. Live server-side rotation is NOT supported without
  a custom NettyReactiveWebServerFactory override (out of scope for MVP).
  Mitigation: use 1-year dev certs; production uses rolling restart + cert-manager.

mTLS AT APPLICATION LEVEL vs SERVICE MESH:
  Service Mesh (Istio/Linkerd): cert issuance (SPIFFE/SVID), rotation, and mutual TLS
  handled by sidecar proxy — zero code change. Operationally superior for production.

  Application-level mTLS (this implementation): every mechanism is explicit in Java code
  — PKCS12 loading, SslContextBuilder, TrustManagerFactory, HttpClient.secure(). This is
  the entire point of the MVP: demonstrate ZT mechanisms transparently before abstracting
  them behind a service mesh.

  Production migration path: strip all mTLS code, let the service mesh sidecar handle it.
  Keep the OBO token mechanism (the service mesh doesn't handle user context delegation).

MANAGEMENT PORT ISOLATION:
  service-a: api=8081 (HTTPS+mTLS), management=9081 (HTTP only)
  service-b: api=8082 (HTTPS+mTLS), management=9082 (HTTP only)
  Docker healthchecks use the management port (HTTP, no client cert needed).
  This avoids the circular problem of needing a client cert to health-check a mTLS endpoint.
