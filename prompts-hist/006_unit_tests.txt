==============================================================
CHAIN OF THOUGHT
==============================================================

WebTestClient vs direct filter unit test:
  WebTestClient requires an HTTP server or @WebFluxTest context which loads beans,
  resolves auto-configuration, and takes seconds to start. For testing a single filter
  in isolation we need none of that. Instead:
    - Instantiate filter directly (constructor injection)
    - Mock PolicyService / UserContextTokenService with Mockito
    - Build a MockServerWebExchange from MockServerHttpRequest (spring-test)
    - Execute filter.filter(exchange, chain) and subscribe via StepVerifier
    - Set the reactive Keycloak JWT context with:
        .contextWrite(ReactiveSecurityContextHolder.withAuthentication(jwtAuth))
  This approach has zero Spring startup cost and is deterministic.

Why mock PolicyService (not AccessPolicyRepository):
  ZteAuthorizationFilter's only dependency is PolicyService. Mocking the repo would:
    (a) require R2DBC connection infra (PostgreSQL running), slowing down CI
    (b) test Spring Data plumbing, not filter logic
    (c) introduce fragile test state (row ordering, DB version)
  PolicyService is the correct mock boundary — it's what the filter actually calls.
  Real integration tests (Testcontainers, planned Stage 5+) will cover the DB layer.

Security context injection pattern (key insight):
  ReactiveSecurityContextHolder.getContext() reads from the Reactor subscriber context.
  In tests, we wrap the StepVerifier subscription with .contextWrite(...) to inject
  a pre-built JwtAuthenticationToken. The filter code is unchanged — only the test
  adds the context. This mirrors exactly what Spring Security does at runtime.
