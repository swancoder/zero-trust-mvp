==============================================================
CHAIN OF THOUGHT (AUTOMATION METHOD SELECTION)
==============================================================

Three automation approaches evaluated:

A) Native realm-export.json + --import-realm flag
   - Built into Keycloak — zero external tooling
   - Runs automatically at container startup
   - Works with dev-file (H2) mode: `docker compose down` clears state → clean reimport on next up
   - Limitation: skips import if realm already exists; requires down (not just restart) to reset
   - VERDICT: Selected for MVP

B) Keycloak Config CLI (codecentric/keycloak-config-cli)
   - Adds Docker init-container; more complex compose YAML
   - Supports variable substitution for multi-env (dev/staging/prod)
   - VERDICT: Rejected — over-engineered for single dev environment MVP

C) Terraform + Keycloak Provider
   - Requires Terraform state, additional binary, CI integration
   - Best for production IaC, but adds toolchain overhead
   - VERDICT: Rejected — incompatible with "zero local dependencies" dev goal

==============================================================
SELF-CRITIQUE
==============================================================

Risk: Hash brittleness in realm-export credentials
Mitigation: zte-admin created with empty credentials[]; password set via kcadm.sh script.
            Avoids invalid PBKDF2 hashes that silently fail import.

Risk: --import-realm skips existing realm (restart ≠ reimport)
Mitigation: Documented. `docker compose down` removes dev-file H2 container data → clean
            state on next `up`. Named volume NOT used for Keycloak data intentionally.

Risk: Realm name mismatch (old default was 'zte', new is 'zte-realm')
Mitigation: Updated application.yml default issuer-uri and jwk-set-uri to point to zte-realm.

Risk: Hardcoded client secret 'zte-gateway-secret' in realm-export.json
Mitigation: Acceptable for dev-only. Documented as NOT for production. Prod should inject via
            KC_BOOTSTRAP env vars or Vault.
