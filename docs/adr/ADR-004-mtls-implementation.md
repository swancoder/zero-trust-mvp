# ADR-004: mTLS Implementation and On-Behalf-Of User Context Delegation

**Status:** Accepted
**Date:** 2026-04-20
**Deciders:** ZTE-Lightweight Architects

---

## Context

Stage 4 adds service-to-service authentication and user context propagation to the ZTE
internal call chain:

```
User ─[Keycloak JWT]──► Gateway ──[mTLS + X-ZTE-User-Context]──► Service A
                                                                        │
                                                          [mTLS + X-ZTE-User-Context forward]
                                                                        ▼
                                                                   Service B
                                                            (validates OBO token + mTLS)
```

Two distinct trust problems must be solved:

1. **Service identity:** How does Service B know the caller is an authorised internal service
   and not an external attacker who guessed the port?
2. **User context delegation:** How does Service B know which end-user the request is on
   behalf of, without re-issuing the original Keycloak JWT (which would be a security liability)?

---

## Decision

**Use mutual TLS (mTLS) for service identity + a signed JWT header (X-ZTE-User-Context)
for user context delegation.**

### Certificate Hierarchy

```
ZTE-CA (self-signed, 1-year dev / 10-year prod root)
├── client.p12      → presented by gateway + service-a as CLIENT cert on outbound calls
├── service-a.p12   → service-a's HTTPS SERVER cert (SANs: service-a, localhost)
└── service-b.p12   → service-b's HTTPS SERVER cert (SANs: service-b, localhost)
truststore.p12      → CA cert only; shared trust anchor for all services
```

All certs are generated by `certs/generate-certs.sh` (OpenSSL + keytool). PKCS12 format.

### Dynamic SslContext Loading (Client Side)

`auth-library/ReloadableSslContextFactory`:
- Loads `client.p12` + `truststore.p12` into a Netty `SslContext` via `SslContextBuilder.forClient()`.
- Holds the context in an `AtomicReference<SslContext>`.
- Reactor Netty's `HttpClient.secure(Consumer<SslContextSpec>)` lambda is evaluated **per TCP
  connection**. Reading `factory.current()` inside the lambda means new connections transparently
  pick up the refreshed context after `refresh()` — no JVM restart, no HttpClient rebuild.
- A `@Scheduled` task in the gateway checks keystore file mtime every 5 minutes and calls
  `refresh()` only when the file has changed (no-op under normal conditions).

### Dynamic SslContext Loading (Server Side)

**Not implemented.** Spring Boot's embedded Netty server builds its SSL context once at startup
from `server.ssl.*` properties. There is no public API to swap the server-side `SslContext`
at runtime without rebuilding the `NettyReactiveWebServerFactory`. The production mitigation
is a rolling restart, which is handled transparently by Kubernetes when cert-manager issues
new certificates.

### OBO Token (X-ZTE-User-Context)

- Compact HMAC-SHA256 JWT created by `UserContextTokenService` (auth-library).
- Claims: `sub` (Keycloak user UUID), `roles` (realm_access.roles), `iss: zte-gateway`, 30s TTL.
- The gateway's `UserContextPropagationFilter` (order `HIGHEST_PRECEDENCE + 200`):
  1. Strips any incoming `X-ZTE-User-Context` and `X-User-Id` headers (prevents injection).
  2. Extracts `sub` and `realm_access.roles` from the validated Keycloak JWT.
  3. Creates and signs the OBO token.
  4. Adds `X-ZTE-User-Context` and `X-User-Id` to the forwarded request.
- Service A forwards `X-ZTE-User-Context` **unchanged** to Service B (delegation, not re-issuance).
- Service B calls `tokenService.validateAndParse()` — if the HMAC signature is invalid or the
  token is expired, it returns 401.

### Management Port Isolation

Services expose their API on an HTTPS/mTLS port and their management endpoints (`/actuator/*`)
on a separate HTTP-only port:

| Service   | API (mTLS) | Management (HTTP) |
|-----------|-----------|-------------------|
| service-a | 8081      | 9081              |
| service-b | 8082      | 9082              |

Docker health checks probe the management port — no client certificate is needed.

## Alternatives Considered

### Option A: Service Mesh (Istio / Linkerd) — rejected for MVP

- **Pros:** Zero-code cert issuance (SPIFFE/SVID), rotation, and mTLS at sidecar level.
  No application changes. Production-grade.
- **Cons:** Requires Kubernetes (or at minimum a VM with sidecar injection). Adds significant
  infrastructure complexity. Hides the ZT mechanisms — defeating the pedagogical goal.
- **Verdict:** Correct choice for production. Rejected for MVP because the entire point is
  to make every mechanism visible and explicit in Java.

### Option B: Pre-Shared Keys / API Keys — rejected

- **Pros:** Zero infrastructure, trivial to implement.
- **Cons:** Not Zero Trust. A static API key is a single point of failure; rotation requires
  coordinating all services simultaneously. No cryptographic service identity.
- **Verdict:** Rejected. Violates the ZT principle of mutual authentication.

### Option C: Asymmetric RS256 for OBO Tokens — deferred

- **Pros:** Gateway signs with its private key; services verify against the gateway's public key
  from a JWKS endpoint. A compromised downstream service cannot forge OBO tokens even if it
  knows its own secret (it doesn't — it only has the public key).
- **Cons:** Requires a JWKS endpoint in the gateway, RSA key management, slightly higher CPU
  cost (RSA vs HMAC).
- **Verdict:** Deferred. Implement as `POST /actuator/obo-jwks` when promoting to production.
  HMAC-SHA256 with a shared secret is acceptable for MVP with a properly secured `ZTE_OBO_SECRET`.

---

## Chain of Thought

1. **Two separate trust channels are necessary.** mTLS proves *which service* is calling
   (transport-layer identity). The OBO token proves *which user* the call is for
   (application-layer delegation). Neither alone is sufficient.

2. **The original Keycloak JWT must not be forwarded.** A downstream service holding the
   user's Keycloak JWT could authenticate as that user to Keycloak or any other service that
   accepts Keycloak tokens. The OBO token is gateway-scoped, short-lived, and carries only
   what downstream services need.

3. **The mTLS lambda-per-connection pattern is the key insight.** Reactor Netty evaluates
   `HttpClient.secure(Consumer<SslContextSpec>)` for each new TCP connection, not at
   `HttpClient` construction time. This makes `AtomicReference<SslContext>` the correct
   pattern for client-side hot-reloading without any request routing change.

4. **Server-side hot-reload is genuinely hard.** Spring Boot doesn't expose a
   `replaceSslContext(SslContext)` API on the running embedded server. A production-grade
   solution requires a custom `NettyReactiveWebServerFactory` subclass that holds a
   `DelegatingSslContextFactory` and calls `server.restart()` or patches Netty's
   `SslContextProvider` directly. For MVP, 1-year certs + rolling restart is the right trade-off.

5. **One shared client cert is appropriate for MVP.** In production, each service would have
   its own SPIFFE identity. For an MVP with two internal services, separating client certs adds
   operational complexity without adding meaningful security boundary.

## Self-Critique

| Risk | Severity | Mitigation |
|---|---|---|
| Shared HMAC secret for OBO tokens | Medium | ZTE_OBO_SECRET env var; rotate on exposure. Upgrade to RS256 for production. |
| Server-side cert rotation requires restart | Medium | 1-year dev certs. Production: cert-manager + K8s rolling update. |
| Shared client.p12 for all internal callers | Low (MVP) | Any service with client.p12 can call service-b. Production: SPIFFE/SVID per service. |
| certs/ directory in project root | Low | .gitignore excludes *.p12, *.key, *.crt. CI/CD runs generate-certs.sh. |
| 30s OBO TTL may expire in slow call chains | Low | Configurable via ZTE_OBO_EXPIRY_SECONDS. |
| `--no-check-certificate` in Docker healthchecks | Negligible | Management port probes are internal to the container (management port is HTTP-only). |

---

## Consequences

- **Positive:** Every ZT trust mechanism is explicit and auditable in Java code.
- **Positive:** `ReloadableSslContextFactory` demonstrates the correct hot-reload pattern
  for Reactor Netty client TLS, applicable to any reactive service.
- **Positive:** `UserContextTokenService` is fully unit-tested without a running server.
- **Negative:** Developers must run `generate-certs.sh` before starting services locally.
- **Negative:** Service-a switches from Spring Web (Tomcat) to Spring WebFlux (Netty) —
  any existing blocking code in controllers must be reviewed.

---

## Future Migration Path

- **RS256 OBO tokens:** Add `GET /actuator/obo-jwks` to gateway; services fetch public key
  on startup and cache it.
- **Service Mesh:** Remove all mTLS code; let Istio/Linkerd sidecar handle cert issuance
  and rotation via SPIFFE. Retain OBO token (mesh doesn't handle user context delegation).
- **Per-service client certs:** Replace `client.p12` with service-specific SPIFFE SVIDs
  (X.509 SVIDs from SPIRE). Service B can then inspect the caller's SVID CN to enforce
  which services are allowed to call it.
